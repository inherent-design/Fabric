# RECURSE ECS Components
*Single Source of Truth for All Component Data Structures*

Components are pure data containers with no behavior. Each component represents a specific aspect of an entity's state. This document defines every component used in RECURSE, serving as the authoritative reference for both documentation and implementation.

---

## Spatial Components

### PositionComponent
```gdscript
class PositionComponent:
    var position: Vector3
    var rotation: Quaternion
    var scale: Vector3 = Vector3.ONE
```

## Component Usage Guidelines

### Component Composition Rules
1. **Minimal Components**: Entities should only have components they actually need
2. **No Redundancy**: Each piece of data should exist in exactly one component
3. **Pure Data**: Components contain no behavior, only state
4. **System Dependencies**: Systems declare which components they read/write

### Common Component Combinations
- **Basic Entity**: PositionComponent, BoundsComponent
- **Physical Object**: + PhysicsComponent, MatterComponent, HealthComponent
- **Destructible Environment**: + VoxelComponent, MaterialInteractionComponent
- **Intelligent Agent**: + AIBrainComponent, SensoryComponent, FactionComponent, MemoryComponent
- **Function User**: + FunctionCapabilityComponent, FunctionExecutorComponent
- **Player**: All of the above + PlayerComponent

### Performance Considerations
- **Critical Components**: PositionComponent, ProcessingLevelComponent (every entity)
- **Heavy Components**: VoxelComponent, MemoryComponent (use sparingly)
- **Optional Components**: EffectComponent, NetworkComponent (only when needed)

This component system provides the complete data vocabulary for RECURSE. Every entity is defined by its component composition, and every system operates by reading and writing these standardized data structures.
